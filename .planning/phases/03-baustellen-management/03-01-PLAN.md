<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | Untrusted input from any HTTP client |
| API → Service | Authenticated user context required |
| Service → Repository | Tenant-isolated data access |
| Repository → Database | Parameterized queries prevent SQL injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | S (Spoofing) | API endpoints | mitigate | JWT validation via auth middleware, AuthenticatedUser extractor |
| T-03-02 | T (Tampering) | Time entries | mitigate | Validate hours > 0, work_date not in future in domain validation |
| T-03-03 | R (Repudiation) | All operations | mitigate | Domain events track all changes with user_id and timestamp |
| T-03-04 | I (Info Disclosure) | Multi-tenant data | mitigate | Every query includes tenant_id filter, no cross-tenant access possible |
| T-03-05 | D (Denial of Service) | API endpoints | accept | No rate limiting in V1, rely on infrastructure |
| T-03-06 | E (Elevation) | Admin operations | mitigate | Role check in service layer, admin-only for site creation/assignment |
| T-03-07 | T (Tampering) | Site assignments | mitigate | Validate user exists in same tenant before assignment |
</threat_model>
