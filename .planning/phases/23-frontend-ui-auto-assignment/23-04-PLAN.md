## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| JWT-auth identity → local user mapping | External identity (`claims.sub`) is mapped to internal DB identity (`users.id`) |
| Preferences API → `user_preferences` writes | Untrusted client intent triggers constrained DB mutations |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-23-GAP-01 | Spoofing | Preferences identity mapping | mitigate | Resolve user through `UserService::get_or_create_from_auth` with tenant context before preferences access; never trust raw JWT subject as local FK target. |
| T-23-GAP-02 | Tampering | `user_preferences.user_id` write path | mitigate | Ensure repository/service receives resolved local `UserId` only; add regression test asserting FK target exists in `users(id)` for same tenant. |
| T-23-GAP-03 | Denial of Service | Repeated 500 errors on toggle | mitigate | Eliminate FK violation path and cover with integration tests to prevent reintroduction. |
| T-23-GAP-04 | Elevation of privilege | Cross-tenant user resolution | mitigate | Keep `TenantContext` tenant ID on lookup/create path; test same keycloak subject across tenants remains isolated. |
</threat_model>
