<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → API | Untrusted input from any HTTP client |
| API → Service | Authenticated user context required |
| Service → Repository | Tenant-isolated data access |
| Repository → Database | Parameterized queries prevent SQL injection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | S (Spoofing) | API endpoints | mitigate | JWT validation via auth middleware, AuthenticatedUser extractor |
| T-04-02 | T (Tampering) | Vehicle/Tool status | mitigate | Status transitions validated in domain layer via can_transition_to() |
| T-04-03 | R (Repudiation) | All operations | mitigate | Domain events track all changes with user_id and timestamp |
| T-04-04 | I (Info Disclosure) | Multi-tenant data | mitigate | Every query includes tenant_id filter, no cross-tenant access possible |
| T-04-05 | D (Denial of Service) | API endpoints | accept | No rate limiting in V1, rely on infrastructure |
| T-04-06 | E (Elevation) | Admin operations | mitigate | Role check in service layer, admin-only for create/update/delete |
| T-04-07 | S (Spoofing) | QR code lookup | mitigate | QR code lookup is tenant-scoped |
</threat_model>
