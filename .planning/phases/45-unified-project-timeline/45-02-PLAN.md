<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| sites API data → timeline UI | Server-provided activity metadata is rendered into the main project context surface |
| timeline preview links → media viewer route | Attachment IDs and filenames flow into deep-link paths and authenticated preview fetches |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-45-05 | Information disclosure | `ActivityFeed` preview rendering | mitigate | Keep `apiClient.getBlob` for protected attachment URLs and do not regress to direct unauthenticated image sources. |
| T-45-06 | Spoofing | timeline metadata display | mitigate | Render timestamps and creator names only from API response fields already scoped by tenant-authenticated endpoints; do not synthesize creator labels client-side beyond the provided data. |
| T-45-07 | Denial of service | timeline-first detail layout | mitigate | Preserve lazy data fetching boundaries already used by `useActivities`, `useTimeEntries`, and `useSiteMaterialHistory`; this phase reorders presentation rather than adding new polling loops. |
| T-45-08 | Tampering | media-viewer deep links | mitigate | Keep attachment navigation built through `buildMediaViewerPath` and existing attachment IDs instead of constructing ad-hoc file URLs in the page component. |
</threat_model>
