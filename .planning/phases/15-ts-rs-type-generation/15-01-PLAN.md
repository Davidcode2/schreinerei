  </read_first>
  <action>
Add ts-rs as a dependency in Cargo.toml with the export feature for type generation:

1. Add to [dependencies]:
```toml
ts-rs = { version = "10", features = ["serde-compat", "chrono-impl", "uuid-impl", "url-impl"] }
```

The `serde-compat` feature ensures ts-rs reads serde attributes (like `rename_all`) for correct type generation. The `chrono-impl` and `uuid-impl` features provide TypeScript type mappings for chrono DateTime and uuid::Uuid types.

Do NOT add ts-rs to dev-dependencies — it must be in regular dependencies because the TS derive macro is used in production code, not just tests.
  </action>

Add ts-rs derive macros to ALL DTOs in the inventory module. For each struct:

1. Add import at top of file:
```rust
use ts_rs::TS;
```

2. For response DTOs (Serialize), change from:
```rust
#[derive(Debug, Serialize)]
pub struct SomeResponse { ... }
```
To:
```rust
#[derive(Debug, Serialize, TS)]
#[ts(export, export_to = "frontend/src/types/generated.ts")]
pub struct SomeResponse { ... }
```

3. For request DTOs (Deserialize), change from:
```rust
#[derive(Debug, Deserialize)]
pub struct SomeRequest { ... }
```
To:
```rust
#[derive(Debug, Deserialize, TS)]
#[ts(export, export_to = "frontend/src/types/generated.ts")]
pub struct SomeRequest { ... }
```

  <action>
Add a step to the CI workflow that verifies generated types are up to date.

Add this step in the test job (after `cargo test` but before completion):

```yaml
- name: Check generated TypeScript types
  run: |
    cargo test --features ts-rs/export
    if ! git diff --exit-code frontend/src/types/generated.ts; then
      echo "ERROR: TypeScript types are out of date!"
      echo "Run: cargo test --features ts-rs/export"
      echo "Then commit the updated frontend/src/types/generated.ts"
      exit 1
    fi
```

This ensures:
1. Any PR that modifies DTOs must also commit updated generated types
2. Type drift is caught in CI, not in production
3. Developers get clear guidance on how to fix the issue

If no CI file exists, create one with this check included in the standard Rust CI workflow.
