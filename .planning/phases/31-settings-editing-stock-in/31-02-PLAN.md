<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| category edit/delete UI → inventory API | Untrusted category form values and destructive actions are sent to protected endpoints |
| inventory list data → card rendering | Category labels are derived from fetched data and shown on overview cards |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-05 | Tampering | `InventorySettingsPage.tsx` category edit form | mitigate | Bind dialog submission to `useUpdateCategory` with explicit `name` and `description` fields only; do not spread arbitrary row objects into mutation payloads. |
| T-31-06 | Repudiation | blocked delete UX | mitigate | Surface 409/conflict failures inline on the affected category row so the user sees that no delete occurred, rather than hiding the result behind a transient generic toast. |
| T-31-07 | Information disclosure | category labels on overview cards | accept | Category names are already visible in inventory filters and belong to the same tenant-scoped inventory dataset; no new cross-tenant surface is introduced. |
| T-31-08 | Denial of service | overview category mapping | mitigate | Reuse the already-fetched `useCategories()` result and derive a local lookup map instead of adding per-card network requests. |
</threat_model>
