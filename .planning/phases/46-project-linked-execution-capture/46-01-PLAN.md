<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → withdrawal dialog | Untrusted user input chooses disposal mode, quantity, notes, and project selection |
| withdrawal request → inventory domain validation | Client payloads must not bypass project-link rules for billable consumption |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-01 | Tampering | `WithdrawMaterial::validate` | mitigate | Reject `disposal: false` requests with `site_id: None` so forged clients cannot create unlinked real consumption rows. |
| T-46-02 | Spoofing | project selector labels in `WithdrawDialog` | mitigate | Display the actual project name plus internal/external cue from API data so workers do not accidentally attribute material to the wrong project type. |
| T-46-03 | Repudiation | withdrawal attribution trail | mitigate | Keep the existing `stock_entries.site_id` linkage on normal withdrawals and reserve `NULL` only for disposal exceptions. |
| T-46-04 | Denial of service | blocked withdrawal UI | mitigate | Use inline validation and disabled submit instead of failed retry loops so missing-project errors are resolved before the request is sent. |
</threat_model>
