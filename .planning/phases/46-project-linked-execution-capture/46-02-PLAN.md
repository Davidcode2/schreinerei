<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → time-entry dialog | Untrusted input chooses work type, hours, date, notes, and optional project |
| time-entry API → site service | Clients must not persist productive work without a project link |
| update patch → merged effective entry state | Partial updates can otherwise clear a project link without changing visible UI state |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-05 | Tampering | `CreateTimeEntry` / `update_time_entry` | mitigate | Enforce productive-type project requirements in backend validation so forged payloads cannot strip `site_id` from billable work. |
| T-46-06 | Repudiation | productive time attribution | mitigate | Preserve `site_id` on both site and workshop work so later reporting can trace labor to a project instead of ambiguous null rows. |
| T-46-07 | Spoofing | time-entry project selector | mitigate | Keep project-type cues in the selector so workshop jobs are intentionally distinguished from external projects during time booking. |
| T-46-08 | Elevation of privilege | update patch semantics | mitigate | Validate the effective merged state in `site_service.rs` before repository write so partial updates cannot bypass the productive-work rule by clearing only one field. |
</threat_model>
