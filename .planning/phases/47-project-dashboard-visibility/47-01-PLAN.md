<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| backend dashboard query → manager dashboard UI | Server-selected project rows determine which projects a manager can see by default |
| manager filter interactions → visible dashboard cards | Client-side filter state changes what the user believes is currently in scope |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-47-01 | Tampering | `get_dashboard_sites` status scope | mitigate | Encode the accepted default status set directly in the SQL query (`planned`, `active`, `completed`) so hidden application-layer filters cannot silently drop completed projects. |
| T-47-02 | Repudiation | dashboard filter state | mitigate | Make status filters explicit in the UI with visible selected-state controls and filter-specific empty copy so managers can see when the list is narrowed intentionally. |
| T-47-03 | Information disclosure | archived project visibility | mitigate | Keep archived projects excluded from the default dashboard query to avoid surfacing stale records in the manager overview while satisfying the phase goal for planned/active/completed work. |
| T-47-04 | Denial of service | mobile dashboard readability | mitigate | Keep the existing compact card grid and use lightweight status controls rather than expanding the dashboard into a heavy reporting surface, preserving readability and quick load/scanning behavior. |
</threat_model>
