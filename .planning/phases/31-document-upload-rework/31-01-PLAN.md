<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → upload API | Untrusted multipart bytes and filenames enter the backend here |
| browser → create activity API | Untrusted attachment IDs and note content cross into tenant-scoped business logic |
| API → database | Attachment linking and activity reads must preserve tenant/site scoping |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | S | create activity attachment linking | mitigate | Resolve all uploaded attachment IDs through tenant_id + site_id before linking; reject IDs outside the caller's scope. |
| T-31-02 | T | multipart upload handler | mitigate | Enforce MIME allowlist and byte-size validation at route/service entry; never trust client filename for storage keys. |
| T-31-03 | I | activity list response | mitigate | Return only attachment metadata from tenant-scoped joins; keep raw storage keys and blob bytes server-side. |
| T-31-04 | D | image/PDF processing | mitigate | Skip thumbnail generation for PDFs and keep existing upload size cap to avoid CPU/memory abuse. |
| T-31-05 | E | attachment association update | mitigate | Require authenticated tenant context and site existence checks before any upload/link/create operation. |
</threat_model>
