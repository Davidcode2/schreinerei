## Interaction Contract

- `InventoryDetailPage` must switch from the legacy history hook to a dedicated enriched-history hook targeting `/history/enriched`; no page-local fetch logic.
- Keep history as its own section on `InventoryDetailPage`, below status/details cards, inside a standard `Card` with `CardHeader` + `CardContent`.
- Each history row is a bordered, rounded item with three visual layers in this order: 1) top row with badge + timestamp + signed quantity, 2) attribution line `von {user_name}`, 3) optional notes / site link metadata.
- Quantity change must remain visually prominent and right-aligned on desktop. Positive stock-in values must show an explicit plus sign; withdrawals must remain visually distinct from stock-in even when both mention quantity.
- For `withdrawn` entries with both `site_id` and `site_name`, render Baustelle as a `Link` to `/sites/${site_id}` using accent blue text and underline-on-hover. If either value is missing, render plain muted text with no empty placeholder.
- Do not style every site reference as a button or badge; it stays an inline text link under the row metadata.
- Notes render only when present. Notes use body size 14-16px muted text and never replace the event label.
- Empty history uses the shared empty-state pattern: dashed/quiet presentation, centered icon container, heading + one-sentence body.
- Loading state may remain lightweight text or skeleton, but error state must use the existing `ErrorState` component contract with retry affordance when the enriched history request fails independently.
- If the row formatting grows beyond comfortable readability inside `InventoryDetailPage`, extract a local presentation component such as `MaterialHistoryFeed` or `MaterialHistoryItem`; do not reuse the sites `ActivityFeed` component.
- German date/time formatting stays `toLocaleString("de-DE")` to match the existing page.
