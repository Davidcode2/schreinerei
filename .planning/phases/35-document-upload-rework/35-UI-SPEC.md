## Phase-Specific Interaction Contract

### 1. Modal scope and layout

- Keep the camera flow separate. Phase 35 changes `CreateNoteModal` into the document composer only; `CameraUploadFlow` remains the dedicated camera/gallery path.
- Use the existing dialog shell pattern from site modals: full width on mobile, `sm:max-w-md` on desktop, `p-6`, `gap-4`, rounded large corners from the shared dialog component.
- Modal title: `Dokumente hinzufügen`.
- Modal description: `Fügen Sie eine Notiz hinzu und laden Sie Bilder oder PDFs in einem Eintrag hoch.`
- Modal content order is fixed:
  1. note textarea
  2. attachment picker
  3. selected-file list
  4. validation / helper text
  5. footer actions

### 2. Note field

- Textarea stays above attachments and remains optional.
- Use 2 visible rows by default, allow vertical growth up to 6 rows before internal scroll.
- Placeholder copy: `Notiz hinzufügen (optional)...`
- Body text uses 14px / 400; helper and validation text use 12px / 600 only for labels and errors.

### 3. Attachment picker

- Replace the current single-photo input with a multi-file picker accepting `image/jpeg`, `image/png`, `image/webp`, and `application/pdf`.
- Enable `multiple` selection.
- Picker label: `Dateien auswählen`.
- Helper copy below picker: `Unterstützt Bilder und PDFs.`
- Do not require drag-and-drop. A button-triggered native picker is sufficient and matches the current mobile-first pattern.
- If a selection contains unsupported files, keep valid files, reject invalid ones, and show one aggregated error message rather than clearing the whole batch.

### 4. Selected-file list

- Render selected files as stacked rows with `gap-2`.
- Each row uses a white card with border, 12px internal padding, and three zones:
  1. 64x64 preview tile
  2. filename + metadata
  3. remove action
- Image files use an object-fit cover thumbnail in a rounded 8px tile.
- PDF files use a neutral tile with `FileText` icon and `PDF` badge instead of a generated preview.
- Filename line: 14px / 600, one line with ellipsis.
- Metadata line: 12px / 400 muted text showing type label (`Bild` or `PDF`). File size is optional and may be omitted if not already available.
- Remove action uses an icon or ghost button, minimum 40x40 target, right-aligned.

### 5. Validation and button states

- The form is valid when either a trimmed note exists or at least one attachment exists.
- The form is invalid only when both are missing.
- Submit button disabled states:
  - pending upload/create request
  - empty note and zero attachments
- Submit button label:
  - idle: `Eintrag speichern`
  - pending: `Speichern...`
- Cancel button stays `Abbrechen` and uses outline styling.
- File-type validation errors appear inline below the picker and may also be mirrored with a toast.
- Network or server failures use toast error feedback with the exact copy from the contract above.

### 6. Feed rendering contract for attachment-backed entries

- Keep the existing activity card shell: icon left, content right, timestamp top-right.
- New document-modal entries with attachments use the file/document icon treatment, not the camera treatment.
- Activity heading rules:
  - note only: `Notiz`
  - note + any attachment(s): `Dokument hinzugefügt`
  - attachments only: `Dokument hinzugefügt`
- Render note text before attachments when note content exists.
- Render attachments below note text in a compact grid:
  - mobile: 2 columns
  - desktop: 3 columns
- Image attachment tiles use rounded preview cards.
- PDF attachment tiles use bordered cards with `FileText` icon, `PDF` badge, and truncated filename.
- Phase 35 does not introduce the fullscreen viewer yet. Attachment previews may be visually interactive, but no fullscreen modal or slug navigation is required until Phase 36.

### 7. Empty, partial, and offline states

- Activity tab empty state copy updates to the contract above so it reflects both notes and documents.
- If attachments are present but their protected blob fetch fails, show the attachment card shell with a fallback file icon and muted text `Vorschau nicht verfügbar` instead of removing the attachment entirely.
- Offline queue behavior remains camera-only for now unless Phase 35 execution explicitly expands offline document queuing. Do not promise offline PDF/document support in the UI copy for this phase.

### 8. Accessibility and interaction details

- Every picker, remove action, and footer button must be keyboard reachable.
- Remove buttons need accessible names such as `Datei entfernen: {filename}`.
- File rows must not rely on color alone to distinguish image vs PDF; keep icon/badge text visible.
- Maintain visible focus ring using the shared `ring` token on picker trigger, textarea, and remove buttons.
