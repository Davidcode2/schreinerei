## Phase-Specific Interaction Contract

### 1. Viewer ownership and routing

- The media viewer is a route-backed overlay, not local-only modal state.
- Canonical route contract: `/sites/:id/media/:activityId/:attachmentId`.
- `App.tsx` owns the route definition; `SiteDetailPage.tsx` owns translating route params into an open viewer state.
- Close behavior is navigation back to `/sites/:id`, preserving browser back/forward behavior for deep links.

### 2. Feed tile interaction

- Every image preview tile and every PDF tile in `ActivityFeed` becomes keyboard-focusable and clickable.
- Tile hit area includes preview + filename card, not image only.
- Hover state: secondary background tint plus 1px accent ring on pointer-capable devices.
- Focus state: visible 2px ring using the shared ring/accent token.
- Existing activity card shell stays unchanged: icon left, heading + timestamp top row, note text below, attachment grid last.
- Feed keeps relative timestamps; viewer uses full absolute timestamp.

### 3. Fullscreen shell

- Viewer must fill nearly the entire screen.
- Mobile: edge-to-edge fullscreen surface (`100dvh`, `100dvw`), no outer radius.
- Desktop: 16px viewport margin, rounded 12px outer corners, still visually fullscreen.
- Layout split:
  - `lg` and above: media pane left/center, fixed metadata sidebar on the right at 320px width.
  - below `lg`: single-column stack with media first and metadata/action panel below.
- Internal padding: 16px mobile, 24px desktop.

### 4. Media rendering rules

- Images render on a darkened viewer backdrop with the image centered inside the main pane using `object-contain`.
- Images must never crop in viewer mode.
- PDFs render in the same viewer shell inside a white document frame using an authenticated blob-backed `iframe` or equivalent embedded browser PDF preview.
- If PDF preview cannot render, keep the viewer open and show a neutral fallback panel with file icon, filename, and the error copy plus the download button.
- Loading state for either type uses a centered skeleton/spinner inside the media pane; do not collapse the viewer while loading.

### 5. Sidebar content and order

- Sidebar order is fixed:
  1. top action row
  2. media title / filename
  3. creator + timestamp block
  4. note section
  5. file details block
- Creator display must use human-readable name, never raw `user_id`.
- Timestamp format in viewer: `DD.MM.YYYY, HH:mm Uhr`.
- Note heading: `Notiz`.
- If note text is missing, render muted body text `Keine Notiz vorhanden.`
- File details block shows at minimum file type and filename; file size may be omitted if not already available from the API.

### 6. Action hierarchy

- Primary action: `Link kopieren` using default/accent button styling.
- Secondary action: `Herunterladen` using outline styling.
- Close action is always visible in the top-right corner with a 44x44px hit target and accessible label `Viewer schlieĂźen`.
- Share success toast: `Link kopiert`.
- Share failure toast: `Link konnte nicht kopiert werden`.
- Download must keep authenticated access; it downloads the protected blob with the original filename instead of exposing a public URL.

### 7. Metadata and titles

- Activity heading inside the feed remains driven by activity type.
- Viewer title uses attachment filename for attachments and `AktivitĂ¤tsfoto` for legacy photo-only entries unless richer backend metadata becomes available.
- If the same activity has note text and attachments, the note belongs to the open media item and is always shown in the sidebar.
- Status-change activities are not viewer targets and must not become clickable.

### 8. Empty, missing, and error states

- Feed empty state remains the Phase 31 contract copy above.
- If a direct-link route references a missing or unauthorized media item, keep the site detail page visible underneath and show an inline viewer error panel with: `Medium konnte nicht geladen werden. - If protected blob fetch fails for a feed preview, keep the fallback tile with `Vorschau nicht verfĂĽgbar`; it must still remain clickable when routing metadata exists.

- Dialog semantics must stay intact when the route-backed viewer is open.
- `Esc` closes the viewer by navigation.
