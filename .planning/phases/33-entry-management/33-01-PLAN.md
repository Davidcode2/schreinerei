<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → activity delete API | Untrusted user input chooses site/activity IDs and attempts destructive actions |
| activity service → PostgreSQL | Trusted backend must enforce tenant, ownership, and cleanup invariants before issuing deletes |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-01 | S/E | `delete_activity` route | mitigate | Resolve the requester to the tenant-local `users.id` in `SiteService` and compare there; never trust client identity hints or frontend ID comparisons. |
| T-33-02 | T | activity + attachment delete SQL | mitigate | Scope repository lookups and deletes by `tenant_id`, `site_id`, and `activity_id` before deleting rows; reject if any scoped lookup fails. |
| T-33-03 | I | activity read DTO | mitigate | Expose only a boolean `can_delete` permission bit instead of leaking authorization rules or requiring raw identity matching on the client. |
| T-33-04 | R | status-change history | mitigate | Reject deletes for `status_change` entries so system workflow history cannot be erased through the new endpoint. |
| T-33-05 | D | attachment cleanup path | accept | Cleanup operates on single activity-scoped rows only; low DoS risk at MVP scale, so no extra throttling beyond existing auth is required. |
</threat_model>
