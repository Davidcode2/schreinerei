# Research Findings

From research/FEATURES.md:
- Hours > 0 validation is table stakes - zero/negative hours make no sense
- Backend already rejects hours <= 0 with validation error
- Frontend should prevent submission before API call

From research/PITFALLS.md (Pitfall 8):
- Input validation feedback only after submit = poor UX
- Users submit multiple times before getting it right
- Client-side validation should mirror backend rules
- Validate on blur + on change (after first blur)

Add validation state to TimeEntryDialog:

1. Add error state:
```tsx
const [errors, setErrors] = useState<Record<string, string>>({})
const [touched, setTouched] = useState<Record<string, boolean>>({})
```

2. Add validation function:
```tsx
const validateHours = (value: number): string | null => {
  if (value <= 0) return "Stunden müssen größer als 0 sein"
  if (value > 24) return "Stunden dürfen 24 nicht überschreiten"
  return null
}
```

3. Update onChange handler:
```tsx
onChange={(e) => {
  const value = parseFloat(e.target.value) || 0
  setHours(value)
  if (touched.hours) {
    const error = validateHours(value)
    setErrors(prev => error ? { ...prev, hours: error } : { ...prev, hours: undefined })
  }
}
```

4. Add onBlur handler to trigger validation:
```tsx
onBlur={() => {
  setTouched(prev => ({ ...prev, hours: true }))
  const error = validateHours(hours)
  setErrors(prev => error ? { ...prev, hours: error } : { ...prev, hours: undefined })
}
```

5. Add error message display below Input:
```tsx
{errors.hours && (
  <p className="text-sm text-destructive">{errors.hours}</p>
)}
```

6. Add red border to input when error exists:
```tsx
<Input
  className={errors.hours ? "border-destructive" : ""}
  ...
/>
```

Do NOT change the quick hours buttons - they set valid values (0.5, 1, 2, 4, 8).
