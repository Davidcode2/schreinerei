<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → sites API | Untrusted create/update payloads choose project type and planning fields |
| sites API → PostgreSQL | Tenant-scoped mutations must stay bound to `TenantContext` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-01 | Tampering | `create_site` / `update_site` payload parsing | mitigate | Parse `project_type` through a closed enum in `src/common/types.rs` and reject invalid values at route entry before repo writes. |
| T-44-02 | Elevation of privilege | `SiteService::create_site` / `update_site` | mitigate | Preserve existing `ctx.is_admin()` guards while threading new fields so broader project editing does not open employee mutation paths. |
| T-44-03 | Information disclosure | `SiteRepository` list/get/update queries | mitigate | Keep every new read/write bound to `tenant_id` in SQL statements and do not add unscoped project lookup helpers. |
| T-44-04 | Repudiation | project-type and planning-field mutations | mitigate | Preserve existing domain-event publication and status/activity audit behavior; new field changes must flow through the same aggregate update path rather than side tables. |
</threat_model>
