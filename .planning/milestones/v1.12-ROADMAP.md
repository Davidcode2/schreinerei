# Milestone v1.12: Architecture Guardrails

**Status:** ✅ SHIPPED 2026-05-04
**Phases:** 41-43
**Total Plans:** 3

## Overview

Hardened the architecture around three high-priority requirements: introduced a transition-safe `projects` module boundary on top of the existing Baustelle implementation, removed manual request-to-tenant-context plumbing from API routes by making tenant context extractable directly, and codified the existing mobile-first standard as an explicit engineering guardrail.

## Phases

### Phase 41: Projects Boundary Alias

**Goal:** Start the `sites` to `projects` architectural transition without breaking runtime routes or storage.
**Depends on:** Nothing
**Plans:** 1 plan

Plans:

- [x] 41-01: Introduce a `projects` module alias that re-exports the current `sites` bounded context and switch the main router composition to that boundary.

**Details:** The runtime remains site-based, but architectural code and future planning can now depend on a `projects` boundary rather than deepening Baustelle-specific naming.

---

### Phase 42: Request Context Extractor

**Goal:** Remove manual `TenantContext::from_auth(&auth)` plumbing from API handlers.
**Depends on:** Phase 41
**Plans:** 1 plan

Plans:

- [x] 42-01: Make `TenantContext` extractable from request extensions and refactor API routes to use it directly.

**Details:** The auth middleware still resolves the authenticated user, but route handlers now receive a request-scoped tenant context directly. IAM service helpers were extended so user bootstrapping can also run from request context instead of raw auth state.

---

### Phase 43: Mobile-First Guardrails

**Goal:** Turn the existing mobile-first behavior into an explicit engineering rule for future milestones.
**Depends on:** Phase 42
**Plans:** 1 plan

Plans:

- [x] 43-01: Create and reference a mobile-first checklist that future user-facing work must satisfy.

**Details:** This phase does not redesign working UI. It captures the current mobile-first baseline as a durable checklist so future milestones do not regress into desktop-first flows.

---

## Milestone Summary

**Key Decisions:**
- Keep the current `sites` implementation stable while introducing a `projects` boundary alias for future architectural work.
- Treat tenant/request context as an extractor concern at the API edge, not as repeated manual conversion inside handlers.
- Execute `ARCH-04` as a guardrail codification milestone because the mobile-first runtime baseline already exists and only needed durable enforcement.

**Issues Resolved:**
- Removed repeated manual request-to-context conversion across IAM, Inventory, Fleet, and Sites route modules.
- Gave future planning and code a `projects` boundary to depend on without forcing a risky rename migration.
- Captured the mobile-first contract in a reusable checklist instead of leaving it implicit.

**Issues Deferred:**
- `ARCH-03` audit coverage standardization remains future work.
- `ARCH-05` object storage migration remains future work.
- The runtime naming and persistence model still use `sites`; only the architectural boundary has started shifting.
