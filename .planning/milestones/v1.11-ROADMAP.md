# Milestone v1.11: Fleet Calendar on Fleet Page

**Status:** ✅ SHIPPED 2026-05-01
**Phases:** 34-36
**Total Plans:** 5

## Overview

Made fleet reservations faster and clearer by moving the calendar directly onto `/fleet`, replacing first-tap instant booking with an explicit two-tap range flow, keeping existing reservations visible during selection, and making `/fleet` the single primary booking path.

## Phases

### Phase 34: Fleet Page Calendar Integration

**Goal:** Users can see and use the fleet calendar directly from the fleet page without losing the existing fleet content below it
**Depends on:** Nothing (first phase of v1.11)
**Plans:** 1 plan

Plans:

- [x] 34-01: Embed the fleet calendar on the fleet page while preserving the current content below it

**Details:** Fleet page renders the reservation calendar as the first major section below the page header, keeps existing fleet tabs and lists usable below it, and reuses the existing calendar implementation instead of adding a second booking path.

---

### Phase 35: Range Selection & Confirmation Flow

**Goal:** Users can create reservations by selecting a date range first, then confirming it in a bottom-positioned modal
**Depends on:** Phase 34
**Plans:** 2 plans

Plans:

- [x] 35-01: Define and test the pure two-tap range-selection contract
- [x] 35-02: Wire calendar interaction, bottom confirmation sheet, and regression coverage

**Details:** First tap starts a pending selection, second tap on the same resource completes a range, same-day double tap works, reverse-order dates normalize earlier-first, and a bottom-positioned confirmation sheet appears only after the second tap. Cancel clears the selection and optional time entry stays opt-in.

---

### Phase 36: Calendar Visibility, Colors & Cleanup

**Goal:** The embedded calendar clearly shows current reservations, uses stable resource colors, and replaces the old primary calendar entry path with regression coverage
**Depends on:** Phase 35
**Plans:** 2 plans

Plans:

- [x] 36-01: Preserve reservation visibility during selection and add deterministic resource colors
- [x] 36-02: Remove the standalone fleet-calendar entry path and lock the embedded route in regressions

**Details:** Existing reservation chips remain visible while selection is active, resource accents are derived deterministically from `resource_type + resource_id`, and the standalone `/fleet/calendar` route and CTA were removed so `/fleet` is the canonical booking experience.

---

## Milestone Summary

**Key Decisions:**
- Embedded the calendar in `/fleet` instead of maintaining a separate primary booking page.
- Switched reservation creation to a two-tap date-range flow rather than immediate first-click booking.
- Used a bottom confirmation sheet so the calendar stays visible during booking confirmation.
- Kept time entry optional instead of forcing start/end times for every reservation.
- Derived resource colors from stable frontend identity hashing instead of backend-managed color state.
- Removed the standalone fleet calendar route once the embedded experience was complete.

**Issues Resolved:**
- Removed the old instant-booking interaction that opened a reservation modal on first tap.
- Preserved existing reservation visibility while adding pending/completed selection states.
- Eliminated the extra `/fleet/calendar` entry path in favor of a single embedded-first booking flow.

**Issues Deferred:**
- Timezone-sensitive `toISOString().split("T")[0]` day-key handling in `CalendarView` remains worth monitoring separately.
- Collision-free uniqueness of resource color accents across arbitrarily large fleets is not guaranteed by current tests.

**Technical Debt Incurred:**
- Resource color tests prove determinism and stability, but not global uniqueness across every possible resource set.
- Calendar day-key derivation still relies on ISO date splitting, which may need a future local-date cleanup if boundary bugs appear.
