# Milestone v1.10: Baustelle Activity Stream Features

**Status:** ✅ SHIPPED 2026-05-01
**Phases:** 30-33
**Total Plans:** 9

## Overview

Enhanced the Baustelle activity stream with separate camera and document upload flows, mixed note-plus-attachment entries, a fullscreen media viewer with deep links, and creator-only entry deletion. The milestone builds on v1.8 photo infrastructure while preserving authenticated media access and tenant-scoped activity behavior.

## Phases

### Phase 30: Camera Upload Flow

**Goal:** Users can capture or select photos directly from the activity stream without opening the document modal.
**Depends on:** Existing v1.8 photo upload infrastructure.
**Plans:** 1 plan

Plans:
- [x] 30-01: Create `CameraUploadFlow` component and wire it into `SiteDetailPage`

**Details:** Added a dedicated camera-first flow with native picker support, image preview, and optional note entry. This keeps the document composer focused on files while preserving the familiar quick photo path for Baustellen use.

---

### Phase 31: Document Upload Rework

**Goal:** Users can create activity entries combining notes and file attachments in any combination.
**Depends on:** Existing document modal.
**Plans:** 3 plans

Plans:
- [x] 31-01: Create backend/storage contract for multi-attachment document entries
- [x] 31-02: Rebuild `CreateNoteModal` into note + multi-attachment document composer
- [x] 31-03: Render attachment-backed document entries in the activity feed

**Details:** Reworked document uploads around upload-first, activity-second orchestration. Users can now submit note-only, attachment-only, or mixed note-plus-image/PDF entries. Feed rendering supports image thumbnails, PDF cards, and preview fallbacks while preserving the separate camera flow.

---

### Phase 32: Media Viewer

**Goal:** Users can view, download, and share media attachments in a fullscreen viewer with direct links.
**Depends on:** Phase 30 and Phase 31.
**Plans:** 3 plans

Plans:
- [x] 32-01: Extend activity API contracts with creator display names for viewer metadata
- [x] 32-02: Add route-backed fullscreen viewer shell, deep links, share, and download actions
- [x] 32-03: Make feed image/PDF tiles open the fullscreen viewer accessibly

**Details:** Added a route-backed media viewer at `/sites/:id/media/:activityId/:attachmentId/:slug?` with metadata sidebar, deep-link copy, protected download, close/back navigation, PDF preview fallback, and mobile/desktop responsive layouts. Follow-up fixes ensured status-change rows remain non-interactive and long strings wrap safely in the feed.

---

### Phase 33: Entry Management

**Goal:** Users can delete their own activity entries with confirmation.
**Depends on:** Existing activity stream entries.
**Plans:** 2 plans

## Milestone Summary

**Key Decisions:**
- Keep camera upload separate from document composition to reduce UI confusion on mobile Baustellen workflows.
- Preserve upload-first, activity-second orchestration so files are stored before business events are created.
- Resolve `creator_name` on the backend instead of comparing Keycloak subjects in the browser.
- Use route-backed viewer overlays so deep links are shareable and close back to `/sites/:id`.
- Expose only `can_delete` from activity APIs so delete permissions remain server-derived.
- Keep `status_change` rows immutable and non-interactive in the viewer surface.

**Issues Resolved:**
- Multi-attachment document model now supports PDFs, images, and mixed note/file submissions.
- Media viewer deep links, metadata, download, and share work for both document attachments and legacy photo entries.
- Entry deletion removes associated attachments and keeps the feed/viewer state consistent.
- Fixed local schema/runtime issues uncovered during verification: optional `thumbnail_key`, creator-name response mapping after create, and status-change/feed wrapping regressions.

**Issues Deferred:**
- Older non-v1.10 verification/UAT gaps and one pending todo were acknowledged at close and recorded in `.planning/STATE.md`.
- Offline photo queue replay remains deferred from v1.8.

**Technical Debt Incurred:**
- Local milestone close proceeded without a dedicated `v1.10` milestone audit file; open artifact deferrals are recorded in `.planning/STATE.md`.
- Runtime migration loading was adopted locally to make new migration files visible during iterative development; this should be revisited for the long-term release strategy.
