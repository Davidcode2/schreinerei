# Milestone v1.8: Activity Feed & Site Status

**Status:** ✅ SHIPPED 2026-05-01
**Phases:** 26-29
**Total Plans:** 11

## Overview

Bring Baustellen to life with status tracking, activity feeds, and linked material history. Users can now change site status through a validated modal, view and add notes in a tabbed activity feed, see material extraction history linked to construction sites, and capture/upload photos with offline support.

## Phases

### Phase 26: Status Change Workflow

**Goal:** Users can change site status through a controlled workflow with validation and audit trail.
**Depends on:** Phase 25 (v1.7 Active Project Context)
**Plans:** 1 combined plan + verification

**Details:** Site status transitions via modal (geplant → aktiv → abgeschlossen) with backend validation, concurrent change protection via optimistic locking, and audit trail through activity entries. "Aktiv" button renamed to "Auswählen" to avoid confusion.

---

### Phase 27: Tabbed Activity Feed

**Goal:** Users can view and add notes in an organized activity feed with clear navigation.
**Depends on:** Phase 26
**Plans:** 1 combined plan + UAT

**Details:** Activity feed reorganization with two tabs (Notizen/Dokumente and Material). Add-note button opens text entry modal. Cursor-based pagination for efficient loading. Each entry shows timestamp and content preview.

---

### Phase 28: Material History Tab

**Goal:** Users can view material extraction history linked to construction sites with full context.
**Depends on:** Phase 27
**Plans:** 3 plans

Plans:
- [x] 28-01: Add backend site-scoped material history endpoint with eager-loaded category/user/site context
- [x] 28-02: Add frontend site history hook + typed contracts for enriched material history rows
- [x] 28-03: Implement Material tab UI rendering and Baustelle navigation link with tests

**Details:** Material tab shows live site-linked extraction history with category, extractor name, quantities, and timestamps. Baustelle names link to site detail pages. Backend uses single eager-loaded query (no N+1).

---

### Phase 29: Photo Upload & Attachments

**Goal:** Users can capture and attach photos to activities with secure storage and offline support.
**Depends on:** Phase 28
**Plans:** 6 plans

Plans:
- [x] 29-01: Add tenant-scoped attachment schema and secure attachment read endpoints
- [x] 29-02: Implement multipart photo upload pipeline with UUID filenames and thumbnail generation
- [x] 29-03: Wire frontend camera/gallery upload flow and activity feed image previews
- [x] 29-04: Add offline photo queueing and reconnect sync processing
- [x] 29-05: Close multipart field mismatch and wire functional photo icon entrypoint
- [x] 29-06: Gap closure — remove duplicate photo activity creation and fix authenticated attachment preview loading

**Details:** Photo uploads via multipart pipeline with MIME validation and UUID-based storage keys. Server-side thumbnail generation. Authenticated blob fetch for image rendering (no unauthenticated URLs). Offline queue with data-URL persistence and reconnect sync. Camera-first modal entry point. Decoupled upload from activity creation.

## Milestone Summary

**Key Decisions:**
- Status change modal with valid transition buttons only
- ActivityFeed displays status changes with arrow icon
- Error handling uses toast notifications with auto-refresh
- "Aktiv" → "Auswählen" to avoid confusion
- Attachment API uses opaque UUID routes — no internal storage keys in public paths
- Attachment lookup returns NotFound for tenant mismatch — prevents cross-tenant disclosure
- Frontend photo flow uploads file first, reuses backend-provided photo_url
- Generate storage keys server-side using UUIDs and MIME-derived extensions
- ApiClient detects FormData and skips JSON serialization for multipart
- Make activity_id nullable — upload stores bytes only, modal createActivity is the single business event path
- Render protected attachment previews via authenticated blob fetch

**Issues Resolved:**
- Multipart field mismatch (file vs photo) fixed across online and offline paths
- Duplicate photo activity creation removed — upload endpoint now stores bytes only
- Unauthenticated image URLs replaced with authenticated blob fetch

**Issues Deferred:**
- Offline photo queue replay (broader offline functionality needed first) — Phase 999.1 backlog

**Technical Debt Incurred:**
- Offline support is partial — only photo uploads queued; full offline mode deferred to v2+
