# Requirements: Schreinerei v1.11 Fleet Calendar on Fleet Page

**Archived:** 2026-05-01
**Core Value:** Mitarbeiter finden alles schnell, Chefs haben den Überblick. Weniger Suchzeit, weniger Fehler, keine vergessenen Bestellungen.

## v1.11 Requirements

### Fleet Page Layout

- [x] **FCAL-01**: User can see the fleet reservation calendar directly at the top of the fleet page — ✅ Delivered
- [x] **FCAL-02**: User can continue using the existing fleet page content below the embedded calendar — ✅ Delivered
- [x] **FCAL-03**: User no longer needs the separate fleet calendar entry point to reach the main booking experience — ✅ Delivered

### Date Range Selection

- [x] **FSEL-01**: User can tap one day in a resource row to start a pending reservation selection — ✅ Delivered
- [x] **FSEL-02**: User can tap a second day in the same resource row to complete a reservation date range — ✅ Delivered
- [x] **FSEL-03**: User can tap the same day twice to create a one-day reservation — ✅ Delivered
- [x] **FSEL-04**: The app sorts the two selected days so the final reservation range always has the earlier date first — ✅ Delivered

### Confirmation and Calendar Feedback

- [x] **FCONF-01**: After the second date is selected, user sees a bottom-positioned confirmation modal showing the selected date range — ✅ Delivered
- [x] **FCONF-02**: User can cancel the confirmation modal to clear the pending selection — ✅ Delivered
- [x] **FCONF-03**: User can confirm the selected range to continue reservation creation — ✅ Delivered
- [x] **FCONF-04**: User can optionally enable time entry and provide start and end times for the booking — ✅ Delivered
- [x] **FCONF-05**: User can still see existing reserved date ranges in the calendar while making a selection — ✅ Delivered
- [x] **FCONF-06**: Each vehicle or machine is shown with a stable unique color in the calendar — ✅ Delivered

## Requirement Outcomes

- **Validated as shipped:** All 13 v1.11 requirements shipped and were verified through phase verification plus milestone audit.
- **Adjusted during delivery:** Stable resource colors were implemented as deterministic client-side identity accents, matching the milestone out-of-scope decision to avoid backend-managed color state.
- **Human follow-up completed:** Bottom-sheet placement, real touch interaction clarity, reservation readability during selection, and color clarity were manually checked and approved before archival.

## Out of Scope

| Feature | Reason |
|---------|--------|
| Drag-to-select calendar interaction | Two-tap range selection is sufficient for this milestone |
| Month view redesign | Milestone is focused on embedding and fixing the existing week-based booking UX |
| Backend-managed resource colors | Stable client-side colors are enough for this milestone |
| Separate mobile and desktop reservation flows | One responsive flow is simpler and less error-prone |
| New reservation backend model | Existing reservation APIs already support the needed date/time data |
