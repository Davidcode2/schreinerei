# Milestone v1.5: Testing & Quality Foundation

**Status:** ✅ SHIPPED 2026-04-30
**Phases:** 12-17
**Total Plans:** 10

## Overview

Establish comprehensive testing strategy, document all issues in existing features, and create an agent QA playbook for future development efficiency.

## Phases

### Phase 12: Backend Domain Tests

**Goal**: Domain layer has comprehensive unit tests for all modules with zero dependencies
**Depends on**: Nothing (first phase)
**Plans**: 2 plans

Plans:
- [x] 12-01: IAM and Inventory domain tests
- [x] 12-02: Sites and Fleet domain tests

**Details:**
- All domain modules (iam, inventory, sites, fleet) have inline unit tests
- Tests use `#[cfg(test)]` attribute in domain files
- Business rules and validation logic covered
- All tests pass in under 1 second (pure unit tests, no DB)

### Phase 13: Agent QA Playbook

**Goal**: Future agent sessions can efficiently validate features using documented procedures
**Depends on**: Nothing (can start immediately)
**Plans**: 1 plan

Plans:
- [x] 13-01: Create QA Playbook, E2E patterns, and ts-rs documentation

**Details:**
- QA-PLAYBOOK.md created with validation procedures
- E2E-PATTERNS.md documents Playwright learnings

### Phase 14: Frontend Test Infrastructure

**Goal**: Frontend has test infrastructure and key component tests for regression prevention
**Depends on**: Phase 13 (uses QA learnings)
**Plans**: 2 plans

Plans:
- [x] 14-01: Vitest setup with test utilities and MSW handlers
- [x] 14-02: Component tests for key dialogs

**Details:**
- Vitest configured with React Testing Library, MSW, and jsdom
- Test utilities exist (render with providers, mock data factories)
- Key components tested (material form, site dialog, vehicle/tool dialogs)

### Phase 15: ts-rs Type Generation

**Goal**: TypeScript types are auto-generated from Rust DTOs, preventing frontend-backend type drift
**Depends on**: Phase 12 (needs stable DTOs)
**Plans**: 1 plan

Plans:
- [x] 15-01: Add ts-rs dependency, derive macros, generate types, CI check

## Milestone Summary

**Key Decisions:**
- Tests inline in domain files for zero friction
- ts-rs v12 instead of v10 (v10 lacks export functionality)
- Vitest over Jest for frontend testing (native Vite integration)
- MSW for API mocking at network level (no axios mocking)
- Domain tests use helper functions for test fixtures

**Issues Resolved:**
- Time booking 400 error root cause identified (hours=0 validation)
- BUG-004 (Fleet Neu button) confirmed FIXED

**Technical Debt:**
- 24 issues documented in ISSUE-BACKLOG.md for v1.6+ roadmap
- E2E tests lack coverage for update/delete operations
- No integration tests with real database (deferred to v1.6)

**Metrics:**
- Backend tests: 116 added
- Frontend tests: 28 added
- E2E data assertions: 6 added
- DTOs with ts-rs: 49
- Issues documented: 24
